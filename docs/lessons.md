Also: structural-queue RVAs are often misplaced (e.g. `getClassMemoryPool` "true start" was actually the already-matched `getModuleNameKey`) — verify an RVA is unclaimed before analyzing.
* Many "size mismatch (110B vs 104B) / misplaced" structural candidates are ICF-folded twins: `getClassMemoryPool@X` and `getModuleNameKey@X` compile to identical 104B so the linker folds them to ONE address; we already match that address as `getModuleNameKey`. The 6 trailing `0xcc` are inter-function padding ghidra counted, not code — and the twin is NOT extra landable bytes (same RVA, one-name-one-address). Don't chase these.
* BFME was built with RTTI off (`/GR-`): only 33 type descriptors exist and they're all exception/STL/locale/ATL types — NO game class (no `File`/`Object`/`LocalFile`). So you cannot read a class vtable from the binary via RTTI; class layout must be triangulated from `call [reg+slot]` sites. Also note matched INI parsers dispatch via function-pointer tables (`call esi`/`edi`), not C++ vtables, so they don't expose the `File` vtable either.
* Confirmed by disassembling `parseDefaultColor` (0x485970): BFME `File::scanString` is vtable slot **9** (`call [eax+0x24]`, `eax=[esi]`=File vtbl) while our ZH build puts it at slot 10 (`0x28`) — so BFME's `File` really does have one fewer virtual before `scanString` (the +1 shift is real, not a misread). To pin the dropped slot you still need BFME's slots for the methods adjacent to `scanString` (`scanReal` our-slot-9, `print` our-slot-11); get them from functions that call those via vtable (not the fn-ptr INI parsers).
* ZH `File` vtable is 16 slots, NOT the 14 you'd count from the visible `virtual` list in `Common/file.h`: `MEMORY_POOL_GLUE_ABC(File)` (GameMemory.h:681) injects two virtuals declared before `open` — slot0 `~File()` (GameMemory.h:683) and slot1 `getObjectMemoryPool()` (GameMemory.h:712). So the real order is 0:`~File`,1:`getObjectMemoryPool`,2:`open`,3:`close`,4:`read`,5:`write`,6:`seek`,7:`nextLine`,8:`scanInt`,9:`scanReal`,10:`scanString`,11:`print`,12:`size`,13:`position`,14:`readEntireAndClose`,15:`convertToRAMFile`. That is why `scanString` is slot 10 (0x28), not 8. BFME has 15 slots (one fewer somewhere in 0..9). WARNING: deleting any virtual in 0..9 shifts `scanString` AND every later slot, so it will BREAK already-matched funcs that use slots >= the deleted one — the net byte delta is uncertain and only the full `./build.sh` can measure it; do not blind-shim this.
* Ghidra 12.1.2 is NOT a shortcut to class vtables here: with RTTI off (`/GR-`) its analyzer only confidently recovers vtables for the string-keyed `FXParticleSystem::*` cluster + the `Module`/`Snapshot`/`Xfer` bases (class/method names are derived from embedded `MODULE_KEY`/`MODULE_NAME` literals returned by `getClass()`, not from a symbol table) — `tools/ghidra/export_vtables.java` dumps them (`reverse/vtables.tsv`, ~204 vtables). `File`/`Object`/`LocalFile`/`RAMFile` vtables are NOT labeled, and free functions stay `FUN_` (no names to pin), so the File/Object/STLport/AsciiString linchpins must still be triangulated from `call [reg+slot]` sites.
* A contiguous `.rdata` scan for "runs of code pointers" does NOT find class vtables here: of the ~1,492 runs of 14-18 code pointers, almost all share the same 10-15-slot suffix (`0xC79D,0x28B87,0x23416,0x15D9D,0x273BD,0x2CB42,0x21A7B,0x351B1,0x7405,0x16B3A`) — these are the INI parser function-pointer DISPATCH tables (`call esi/edi`), not C++ vtables, so the scan conflates them and can't isolate `File`/`Object`. The only reliable way to read a specific class vtable is constructor-anchored: find a `LocalFile`/`RAMFile` ctor that does `mov [this], imm32` (imm32 = the vtable in `.rdata`), then dump its slots. (Also: a batched `struct.unpack_from('<IIII',d,b+8)` for PE section headers reads `(VirtualSize, VirtualAddress, SizeOfRawData, PtrToRawData)` — assign in that order, not `(vrva,vsize,...)`, or every downstream RVA/VMA is silently wrong.)
* BFME's import IAT (FirstThunk) is BOUND (slots hold real `0x7C…`/`0x76…` addresses, not name RVAs), so to map an IAT slot to a name read the NAME from the parallel ILT (OriginalFirstThunk) and the SLOT VMA from the IAT (`IB + FirstThunk + i*4`); reading the name from the IAT entry itself yields nothing. Also BFME memory-maps files: file open is funneled through one free helper `FUN_00e3c63b` (calls `CreateFileA/W`,`GetFileSize`,`CreateFileMappingA`,`MapViewOfFile`) — so `LocalFile::open` is a vtable slot that CALLS that helper rather than inlining Win32, meaning the `File` vtable is anchored by finding callers of `FUN_00e3c63b` (one level up), not by CreateFileA sites. Gotcha: `re.finditer(re.escape(pat), …)` on a bytes regex that contains a `[…]` char-class silently matches 0 (re.escape turns `[` into a literal) — never re.escape an already-built bytes regex.
* Anchoring the `File` vtable via `FUN_00e3c63b` does NOT work either: its 15 callers are all free loader functions, none a vtable slot — because BFME reads from `.BIG` archives, so `LocalFile::read/seek/size` operate on an in-memory archive index and do NOT call `ReadFile/SetFilePointer/GetFileSize` (those Win32 calls live only in the open/mmap helper and higher loaders). So the `File` vtable's slots point to custom `FUN_` (archive-layer) code, not OS imports — the only remaining way to read it is constructor-anchored (`mov [this], imm32` in a `LocalFile`/`RAMFile` ctor) or by first naming `scanString`'s body (slot 9) and searching `.rdata` for it.
* Anchoring `File` via `File::print/write` is ALSO dead: MSVCR71 `fprintf`(0x13593c0)/`fwrite`(0x13593f4) have vtable-slot callers, but the only `fprintf+fwrite` slot is `FUN_00d93a40`, a len-1 `.rdata` singleton (a free helper/callback, not a class vtable) — same funnel pattern as `FUN_00e3c63b`; the other 5 slots call only `sprintf` (AsciiString formatting). So no import reaches a `File` slot. Bonus anchor: BFME `AsciiString`/`StringBase<char>` vtable is at **0x125f114** (5 slots: 0x887b60,0x888f90,0x888b50,**0x888bc0**,0x888b80=sprintf format) — confirms 0x888bc0 is a virtual in the AsciiString base and gives a fixed point for the AsciiString/StringBase drift. `build/find_file_print.py` reproduces this scan.
